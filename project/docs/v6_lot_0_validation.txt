SkateGame Engine Prototype - Validation Lot 0 V6

Role du document
Ce document fige les decisions de cadrage qui doivent etre considerees
comme actees avant le debut de l'implementation de la V6.

Statut
Lot 0 valide.

Date de validation
2026-04-15

Rappel du role du lot 0
Le lot 0 sert a verrouiller les decisions qui evitent les demi-retours
pendant le developpement.

Decisions validees

1. Nature de la V6
La V6 est une version de politiques de match et de presets.
Elle n'est pas encore la version d'une grosse vague de nouvelles regles fines.

2. Changement d'architecture
La V6 demande un petit changement d'architecture, mais pas une refonte.
Le changement doit rester additif.

3. Emplacement du chantier V6
La V6 ne demande pas de nouveau dossier.
Le chantier doit surtout faire evoluer `project/config/`.

4. Presets V6
La V6 introduit des presets officiels internes au moteur.
Ils sont definis en code.

5. Presets utilisateur
La V6 ne prevoit pas encore :
- creation de presets utilisateur
- sauvegarde de presets utilisateur
- chargement de presets utilisateur
- import/export de presets

6. JSON de presets
La V6 ne prevoit pas encore de stockage JSON pour les presets.
Si un systeme de presets externes apparait plus tard,
il viendra dans une phase ulterieure du projet.

7. Objets de configuration V6
La V6 doit reposer sur la separation suivante :
- `MatchParameters`
- `RuleSetConfig`
- `MatchPolicies`
- `MatchPreset`
- `PresetRegistry`

8. Role de `RuleSetConfig`
`RuleSetConfig` doit rester centre sur les regles pures.
Il ne doit pas absorber les choix structurels du flow.

9. Role de `MatchPolicies`
`MatchPolicies` doit porter les choix structurels du match.
Exemples V6 cibles :
- ordre initial
- rotation de l'attaquant
- ordre des defenseurs

10. Role de `MatchPreset`
`MatchPreset` decrit un preset officiel en memoire.
Il ne gere pas lui-meme des JSON.

11. Role de `PresetRegistry`
`PresetRegistry` centralise les presets officiels disponibles.

12. Persistence V6
Le save/load d'une partie V6 doit sauvegarder la configuration effective du match,
et pas seulement un nom de preset.

13. Interfaces V6
CLI et GUI doivent proposer et afficher les presets officiels,
sans devenir un chantier UX lourd.

14. Sujet secondaire V6
Le `join between turns` est compatible avec la V6,
mais seulement comme lot secondaire apres le coeur V6.

15. Premier scope de reconfiguration
Si ce lot secondaire est pris, le premier cas officiel vise est :
`one_vs_one -> battle`
entre deux tours uniquement.

16. Sujet `phases.py`
Le fichier `core/phases.py` n'avait pas d'utilite concrete
dans l'architecture actuelle et a ete supprime.
La V6 ne repose pas sur une separation globale en phases d'attaque et de defense.

Ce que le lot 0 debloque
Le chantier peut maintenant entrer dans le lot 1 :
definition precise des objets V6 dans `config/`.

Questions qui restent ouvertes apres le lot 0
Ces points restent a definir dans le lot 1,
mais sans remettre en cause les decisions de cadrage :
- forme exacte de `MatchPolicies`
- champs exacts de `MatchPreset`
- place exacte de `preset_name` dans `MatchParameters`
- nombre final de presets officiels V6

Definition pratique du lot 0 termine
Le lot 0 est considere termine si :
- la direction V6 est claire
- les non-objectifs sont explicites
- les objets de haut niveau sont identifies
- l'emplacement du chantier est decide
- les sujets deferes sont assumes comme tels

Conclusion
Le lot 0 V6 est valide.
La prochaine etape logique est le lot 1 :
figer la forme des objets de configuration V6.

SkateGame Engine Prototype - Validation Lot 3 V6

Role du document
Ce document resume ce qui a ete effectivement livre pour le lot 3 de la V6.

Statut
Lot 3 implemente.

Date de validation
2026-04-15

Rappel du role du lot 3
Le lot 3 sert a faire vivre les policies V6 dans le moteur reel,
sans dupliquer artificiellement les modes.

Livraisons effectives

1. Injection des policies dans les modes
`BaseMode` accepte maintenant un objet `MatchPolicies`.
Chaque mode dispose donc de ses policies propres.

2. Helpers structurels dans `BaseMode`
La logique commune suivante a ete centralisee :
- construction des defenseurs a partir de `turn_order`
- inversion eventuelle de l'ordre des defenseurs
- calcul du prochain attaquant selon la rotation supportee

3. Evolution de `OneVsOneMode`
`OneVsOneMode` s'appuie maintenant sur les helpers communs du `BaseMode`
pour :
- construire les defenseurs
- choisir le prochain attaquant

Son comportement officiel reste inchange sur le flow supporte.

4. Evolution de `BattleMode`
`BattleMode` tient maintenant compte des policies V6 pour :
- l'ordre initial
- l'ordre des defenseurs

Concretement :
- `InitialTurnOrderPolicy.RANDOMIZED` conserve le comportement V5
- `InitialTurnOrderPolicy.FIXED_PLAYER_ORDER` permet un ordre initial stable
- `DefenderOrderPolicy.REVERSE_TURN_ORDER` inverse l'ordre des defenseurs

5. Evolution de `ModeFactory`
`ModeFactory.create(...)` accepte maintenant les `MatchPolicies`
et les transmet au mode cree.

6. Evolution de `GameEngine`
`GameEngine` charge maintenant le mode via :
- `mode_name`
- `policies`

Le meme principe est applique au `load_game()`.

Resultat structurel
Le moteur n'est plus seulement capable de choisir un mode.
Il commence a choisir une variante structurelle de ce mode.

Tests ajoutes
`tests/unit/test_mode_policies.py` couvre maintenant :
- transmission des policies par la factory
- ordre initial fixe en `battle`
- ordre inverse des defenseurs en `battle`

Verification effectuee
- `tests/unit/test_mode_policies.py`
- `tests/unit/test_game_flow.py`

Resultat
Ces tests passent.

Ce que le lot 3 ne fait pas encore

1. Persistence complete V6
Les nouveaux champs V6 ne sont pas encore pleinement serialises.
Ce travail appartient au lot 4.

2. Interfaces V6
CLI et GUI ne pilotent pas encore les presets.
Ce travail appartient au lot 6.

3. Matrice de tests multi-presets
Le moteur comprend deja les policies,
mais la vraie couverture multi-presets reste a consolider plus tard.

Decision implicite importante
Le lot 3 branche deja une partie utile des `MatchPolicies`,
mais pas encore toute la richesse future de la V6.

En particulier :
- `attacker_rotation` n'introduit pas encore de variante supplementaire
- la politique de fin de match reste encore largement portee par l'existant

Ce que le lot 3 debloque
Le chantier peut maintenant entrer dans le lot 4 :
persistence des nouveaux objets V6.

Conclusion
Le lot 3 a fait franchir au projet le passage important entre
"configuration V6 declaree" et "configuration V6 reellement comprise par le moteur".

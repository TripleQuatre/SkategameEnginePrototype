SkateGame Engine Prototype - Validation Lot 8 V7

But du document
Ce document fige les decisions du lot 8 sur le realignement minimal
des interfaces et de la creation de partie avec la logique V7.

Statut du lot
Lot 8 considere comme valide.

Rappel du role du lot 8
Le lot 8 ne cherche pas a lancer un chantier UX autonome.
Il garantit que les interfaces CLI/GUI restent alignees
avec la V7 tout en conservant une facade produit simple.

Conclusion generale
La V7 garde une creation de partie volontairement simple.
Le `mode` reste central cote interface,
mais il est traduit vers la verite moteur effective du match.

La principale complexite interface du lot 8
concerne surtout les transitions de roster
et leurs effets structurels implicites.

Decision 1. Le `mode` reste central cote interface

Decision retenue
Cote produit, le `mode` reste un point d'entree central
de la creation de partie,
meme s'il n'est plus un centre structurant du moteur.

Pourquoi
Le produit doit rester simple et lisible
pour la creation de partie.

Effet attendu
La V7 conserve une facade utilisateur claire
sans exposer directement la structure interne du moteur.

Decision 2. La creation de partie V7 reste tres simple

Decision retenue
La creation de partie conserve les regles produit suivantes :
- minimum 2 joueurs
- 2 joueurs => `1v1`
- 3+ joueurs => `battle`
- pas de choix du sport
- pas de setup avance expose

Pourquoi
Le nombre de parametres reste encore raisonnable
meme avec `attack_attempts`.

Effet attendu
La V7 protege la simplicite actuelle du produit
pendant que le moteur se renforce.

Decision 3. L'interface traduit le setup produit
vers la verite moteur

Decision retenue
Les interfaces CLI/GUI ne construisent pas directement
la configuration moteur effective.
Elles passent par une traduction explicite
du setup produit vers la config reelle du match.

Pourquoi
Cette separation prepare proprement :
- la sauvegarde de setup
- le partage futur de setup
- l'evolution des presets

Effet attendu
Le lot 8 reste coherent avec la separation
entre produit et moteur deja validee.

Decision 4. La principale complexite interface V7
concerne les transitions

Decision retenue
La complexite UX principale du lot 8 concerne surtout :
- l'ajout de joueur
- le retrait de joueur
- l'effet structurel implicite sur `1v1` / `battle`

Precision retenue
Le choix de parametres comme :
- `letters`
- `attack_attempts`
- `defense_attempts`
ne constitue pas encore, a ce stade,
une complexite majeure cote produit.

Pourquoi
Le lot 8 doit rester focalise
sur les vrais points sensibles de l'interface V7.

Effet attendu
Le chantier interface reste sobre
et correctement cible.

Decision 5. Le lot 8 vise un alignement minimal,
pas une refonte UX

Decision retenue
Le lot 8 cherche seulement a realigner proprement
les interfaces CLI/GUI avec la logique V7.

Il ne vise pas :
- une grosse refonte d'interface
- une experience de setup tres enrichie
- un chantier UX autonome

Pourquoi
La priorite reste la coherence moteur,
pas une transformation produit lourde.

Effet attendu
La V7 garde un branchement produit propre,
leger et suffisant.

Decision 6. La fin de V7 pourra ouvrir
quelques setups supplementaires

Decision retenue
Sans creer un setup avance lourd,
la fin de V7 pourra proposer
quelques nouveaux setups
en jouant sur `attack_attempts`.

Pourquoi
La nouvelle famille `Attack`
commence a rendre possibles
des variations produit lisibles
sans exploser la complexite.

Effet attendu
La V7 pourra montrer un peu plus de richesse visible
sans perdre sa simplicite globale.

Coherence avec les lots precedents
Le lot 8 est aligne avec :
- le lot 0 sur la place du `mode`
- le lot 1 sur la separation `application` / `match`
- le lot 2 sur la traduction setup produit -> config moteur
- le lot 5 sur la place centrale des transitions de roster
- le lot 6 sur l'arrivee de `attack_attempts`

Ce qui est officiellement hors lot 8
- le choix du sport
- une grosse refonte UX/UI
- un setup produit tres parametrique
- une nouvelle arborescence d'interfaces complete
- un chantier graphique autonome

Definition de succes du lot 8
Le lot 8 est reussi si :
- la creation de partie reste simple
- le `mode` reste lisible cote produit
- les interfaces ne reconstruisent pas la verite moteur
- les transitions restent le principal point sensible interface
- la V7 peut exposer quelques setups supplementaires en fin de version
  sans perdre sa sobriete

Conclusion
Le lot 8 valide que la V7 peut changer profondement
son architecture interne sans casser
la simplicite produit actuelle.
Les interfaces restent legeres,
mais correctement alignees avec la nouvelle logique du moteur.

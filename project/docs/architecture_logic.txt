SkateGame Engine Prototype - Architecture V8

Objectif
Cette documentation decrit l'architecture reelle du projet apres la V8.
La V7.x a recentre le runtime autour de `MatchConfig` et du dossier `match/`.
La V8 a ajoute une nouvelle couche metier autonome : `dictionary/`.

Principe general
Le projet repose maintenant sur une architecture articulee autour de trois
centres clairs :

- `core/` pour l'etat et les structures fondamentales
- `match/` pour le runtime de partie
- `dictionary/` pour la definition canonique des tricks

Lecture simplifiee

Interfaces -> Controller -> Application Session -> Match Runtime
                                           -> Validation
                                           -> Dictionary
                                           -> Persistence
                                           -> Core
                                           -> Config

Idees fortes de la V8
- `GameState` reste la source de verite du runtime courant
- `MatchConfig` est la source de verite de configuration
- `match/` porte l'orchestration metier reelle d'une partie
- `dictionary/` dit ce qu'est un trick valide et quelle est son identite
- `validation/` bloque les actions et etats incoherents avant mutation
- `persistence/` sait conserver la configuration et le runtime complet
- `SnapshotHistory` reste la base de l'undo

Blocs principaux

1. `core/`
- structures fondamentales du moteur
- `GameState`, `Player`, `History`, `Snapshot`, enums runtime

2. `config/`
- modeles de configuration de match
- presets
- traduction setup -> runtime config
- regles fines (`Uniqueness`, `Repetition`)

3. `dictionary/`
- domaine autonome du dictionnaire de tricks
- modele canonique des segments, combos et sorties
- premiere implementation concrete : `inline_primary_grind`

4. `match/`
- coeur du runtime de partie
- structure du match
- flow d'attaque
- flow de defense
- scoring
- victory
- transitions de roster
- regles metier de tour

5. `application/`
- session applicative de haut niveau
- point d'entree utile pour controller, interfaces et tests

6. `controllers/`
- relais mince entre interfaces et session applicative

7. `validation/`
- validation de config
- validation d'action
- validation d'etat

8. `persistence/`
- serialization du runtime, de la config, de l'historique
- save/load

9. `interfaces/`
- CLI et GUI utilitaires
- saisie assistee branchee sur `dictionary/`

10. `tests/`
- unitaires
- integration
- interfaces

Decisions V7.x / V8 importantes

Configuration
- le runtime est pilote par `MatchConfig`
- `MatchParameters` reste une vue de compatibilite, pas le centre du moteur

Structure de match
- le vieux centre `modes/` n'est plus le coeur actif
- la structure vit maintenant dans `match/structure/`

Tricks
- `current_trick` reste present pour la compatibilite
- les donnees canoniques V8 vivent en parallele :
  - `current_trick_data`
  - `validated_trick_data`
  - `failed_attack_trick_data`

Regles fines V8
- `Uniqueness` lit les tricks valides
- `Repetition` lit les tricks lances puis rates
- ces regles comparent l'identite canonique du dictionnaire, pas seulement du
  texte libre

Undo
- snapshots complets de state + config
- restauration complete sans suppression manuelle d'evenements

Save / Load
- une sauvegarde transporte `match_config` + `game_state`
- l'undo local est reinitialise apres chargement

Etat actuel de la V8
- runtime stabilise autour de `match/`
- dictionnaire autonome et branche au moteur
- premiere discipline/profile utilisable : `inline / inline_primary_grind`
- saisie assistee CLI et GUI
- `Uniqueness` et `Repetition` actives
- suite de tests complete verte

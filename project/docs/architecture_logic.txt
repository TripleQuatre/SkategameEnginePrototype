SkateGame Engine Prototype - Architecture V5

Objectif
Cette documentation decrit l'architecture reelle du projet apres la V5.
Le but de la V5 etait d'etendre le moteur vers le multijoueur sans casser
la robustesse acquise en V4.

Principe general
Le projet repose sur un moteur central testable :

Interfaces -> Controller -> Engine -> Modes
                         -> Rules
                         -> Validation
                         -> Persistence
                         -> Core

Idees fortes de la V5
- `GameState` reste la source de verite principale.
- `History` garde la trace brute du match et reconstruit des tours lisibles.
- `SnapshotHistory` sert a l'undo par restauration complete d'etat.
- les noms d'evenements restent centralises via `EventName`.
- les invariants d'etat sont verifies explicitement.
- la structure d'un match n'est plus codee en dur dans `game_flow`.

Blocs principaux

1. `core/`
- contient les structures fondamentales du moteur
- `GameState`, `Event`, `History`, `Snapshot`, `Player`, `EventName`, `Phase`

2. `modes/`
- definit la structure d'une partie
- initialise le match
- construit les defenseurs
- determine le prochain attaquant

3. `rules/`
- contient la logique metier pure
- pas d'orchestration de match

4. `engine/`
- orchestre la partie
- applique les transitions
- gere l'undo et le save/load au niveau moteur

5. `validation/`
- verifie config, etat et actions
- protege les invariants les plus importants

6. `controllers/`
- relais fin entre interfaces et moteur
- expose une API simple aux interfaces

7. `persistence/`
- serialization, sauvegarde et rechargement de partie

8. `interfaces/`
- CLI et GUI utilitaires
- pas de logique metier critique

Decisions V5 importantes

Etat du tour
- tour ouvert : l'attaquant a la main et `current_trick is None`
- tour engage : `current_trick is not None`
- tour termine : le tour a produit un resultat terminal

Rotation
- `turn_order` porte l'ordre persistant du match
- les modes l'interpretent pour la rotation
- les joueurs elimines peuvent rester dans `turn_order`

Tricks
- `validated_tricks` contient uniquement les tricks definitivement consommes
- `start_turn(trick)` engage le trick du tour, sans le consommer
- le trick est consomme a la fin terminale du tour :
  - `TURN_FINISHED`
  - `GAME_FINISHED`

Tour rate
- `cancel_turn(trick)` produit `TURN_FAILED`
- un tour rate reste un evenement metier normal de la partie

Undo
- l'undo repose uniquement sur des snapshots complets
- un snapshot est pris avant chaque action mutante exposee par le moteur
- l'undo restaure un etat entier valide, sans suppression manuelle d'evenements
- l'undo reste possible apres une fin de partie dans la session courante

Save / Load
- le moteur sauvegarde et recharge un `GameSave`
- un `GameSave` contient :
  - `match_parameters`
  - `game_state`
- un chargement remet l'historique d'undo a zero
- le mode est recharge a partir de la sauvegarde

Etat actuel de la V5
- moteur multi-mode stabilise
- `battle` present
- fin de partie consultable
- save/load multi-mode
- interfaces testees sur les parcours critiques

Points volontairement laisses pour la suite
- variantes de rotation configurables
- systemes de points alternatifs
- gestion d'equipes
- outillage encore plus riche de simulation de configurations

V9.2 Harness Usage

Objective
The GUI harness can run in two ways:
- integrated through pytest for critical reference scenarios
- manually through a dedicated CLI entry point for focused local runs

Integrated tests
- reference scenarios are covered by
  `tests/interfaces/test_harness_reference_scenarios.py`
- this keeps the V9.2 harness connected to the normal test suite

Run one scenario manually

python -m harness.cli --scenario harness/scenarios/reference/preset_one_vs_one_smoke.yaml

Run the reference suite manually

python -m harness.cli --reference-suite

Run one bounded semi-random scenario

python -m harness.cli --random-bounded --seed 5 --max-random-steps 18

Semi-random replay
- the same seed produces the same generated scenario
- reports keep the seed and generated sequence
- the exploration is bounded to safe GUI actions and known stable fragments
- this is not free-form fuzzing

Optional flags
- `--show-window`
  keeps the Tk window visible during the run
- `--saves-dir <path>`
  controls where save-game artifacts are written during harness runs

Reference scenarios
- `preset_one_vs_one_smoke.yaml`
- `custom_battle_smoke.yaml`
- `history_and_setup_details_navigation.yaml`
- `undo_to_setup_regression.yaml`
- `save_smoke.yaml`

Other scenario families
- `harness/scenarios/stress/`
  for longer deterministic stress sequences
- `harness/scenarios/regression/`
  for named bug regressions kept permanently in the harness library

Stress matrix note
- V9.3 also introduces an explicit `stress_matrix`
- this is not a giant automatic matrix
- it is a curated set of useful `preset x stress scenario` combinations chosen
  for mechanical coverage and easier diagnosis

Expected output
- each scenario prints `[PASS]` or `[FAIL]`
- failures print the failing step, error message, expected value, observed value,
  and artifact path when available
- structured reports now also keep:
  - scenario context (`tags`, `family`, `setup`, scenario path, step count)
  - direct failed-step context (`name`, `action`, `target`, `expect`)
  - step-by-step expectations
  - visible-state transitions between steps
  - compact deltas on buttons, key texts, score cells and dropdown state
  - compact visible-state highlights adapted to the current view
    - setup: order preview, synergy feedback, summary
    - match: phase/trick/attempts and key button states
    - history: row count and first row
    - setup details: first structured lines of the setup body

Stress reading note
- this richer reporting is especially useful on long stress or regression runs
- the important question is no longer only "which step failed"
- it is also "under which config did it fail" and "what changed right before it"

Boundary reminder
- the harness remains a tooling layer
- it drives the GUI through instrumented widgets
- it does not become a dependency of the game engine

Documentation V8 - Logique des donnees

But
Ce document decrit la logique de donnees importante du moteur, independamment
des interfaces.

Sources de verite
- `GameState` : etat courant du runtime
- `MatchConfig` : configuration runtime du match
- `History` : trace des evenements
- `Snapshot` : image restaurable pour l'undo
- `GameSave` : enveloppe persistable du match
- `ConstructedTrick` : identite canonique d'un trick cote dictionnaire

Distinctions cles

1. Etat courant
- ce que le moteur est en train de vivre
- porte par `GameState`

2. Configuration
- ce qui pilote structure, scoring, victory et regles fines
- porte par `MatchConfig`

3. Historique
- ce qui s'est produit
- porte par `History`

4. Snapshot
- ce qui peut etre restaure rapidement
- porte par `Snapshot` et `SnapshotHistory`

5. Sauvegarde
- ce qui peut etre stocke et recharge
- porte par `GameSave`

6. Identite canonique d'un trick
- ce que `Dictionary` considere comme le vrai trick
- porte par les objets de `dictionary/`

Donnees sensibles du state
- `turn_order`
- `attacker_index`
- `defender_indices`
- `current_defender_position`
- `attack_attempts_left`
- `defense_attempts_left`
- `current_trick`
- `current_trick_data`
- `validated_tricks`
- `validated_trick_data`
- `failed_attack_trick_data`
- `history`

Semantique actuelle des tricks
- un trick saisi passe idealement par `dictionary/`
- le texte brut peut encore exister pour compatibilite
- l'identite canonique V8 vit en parallele
- un trick valide alimente `validated_trick_data`
- un trick lance puis rate alimente `failed_attack_trick_data`

Semantique V8 de l'historique des tricks
- `Uniqueness` lit les tricks valides
- `Repetition` lit les tricks lances puis rates
- les deux travaillent sur une identite canonique si elle existe

Pourquoi c'est important
Cette separation permet de garder coherents :
- la saisie assistee V8
- `Uniqueness`
- `Repetition`
- l'undo
- le save/load
- les futures regles comme `Banned Tricks`

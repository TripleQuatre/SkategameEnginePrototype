# 📁 Dossier `engine/`

## 🎯 Rôle global

Le dossier `engine/` est le **cœur actif du moteur**.

C’est ici que :

* le jeu avance
* les règles sont appliquées
* les décisions sont prises
* les événements sont produits

👉 Si `core/` = données
👉 alors `engine/` = **comportement**

---

## 📄 `game_engine.py`

### Rôle

Point d’entrée principal du moteur.

Responsabilités :

* créer l’état initial (`GameState`)
* charger le mode de jeu
* initialiser les dépendances (rules, validators, flow)
* exposer une API simple au controller

### Position dans l’architecture

* appelé par `GameController`
* encapsule `GameFlow`
* sert de façade

### Évolution probable

* gestion de plusieurs modes
* injection dynamique de règles
* éventuellement point d’entrée pour charger un état (persistance)

### Améliorations possibles

* méthode `_load_mode()` typée (déjà corrigé)
* futur support multi-mode plus flexible

---

## 📄 `game_flow.py`

### Rôle

C’est le **cerveau du jeu**.

Responsable de :

* démarrer une partie
* lancer un tour
* gérer les phases
* orchestrer la défense
* gérer les transitions
* produire les événements

### Position dans l’architecture

* utilisé par `GameEngine`
* utilise :

  * `TurnResolver`
  * `EndConditions`
  * `ActionValidator`
  * `RulesRegistry`

### Évolution probable

* gestion de modes plus complexes
* enrichissement des phases
* gestion multi-joueurs plus avancée

### Améliorations possibles

* aucune urgente
* attention future à ne pas le surcharger

👉 C’est un fichier critique à garder lisible

---

## 📄 `turn_resolver.py`

### Rôle

Gère la **résolution d’une tentative de défense**.

Responsabilités :

* déterminer si une tentative réussit ou échoue
* appliquer le score si nécessaire
* déléguer les règles de scoring

### Position dans l’architecture

* appelé par `GameFlow`
* utilise `ScoringRules`

### Évolution probable

* prise en charge de règles plus complexes
* gestion de cas multi-défenseurs plus riches

### Améliorations possibles

* aucune nécessaire pour la V3
* le découplage avec `GameFlow` est bon

---

## 📄 `end_conditions.py`

### Rôle

Gère les **conditions de fin de partie**.

Responsabilités :

* détecter si un joueur est éliminé
* déterminer si la partie est terminée
* identifier le gagnant

### Position dans l’architecture

* utilisé par `GameFlow`
* dépend de `GameState`

### Évolution probable

* support de modes avec conditions différentes
* variantes de victoire (points, manches…)

### Améliorations possibles

* rien d’urgent
* bien isolé, donc facile à faire évoluer

---

## 🧠 Conclusion du dossier `engine/`

Le dossier `engine/` est :

* bien découpé
* lisible
* modulaire

👉 C’est un point fort de la V3.

---

## 🔎 À retenir

* `game_engine.py` = point d’entrée / façade
* `game_flow.py` = orchestration globale
* `turn_resolver.py` = résolution des actions
* `end_conditions.py` = fin de partie

---

## ⚠️ Point d’attention futur

Le seul vrai risque ici est :
👉 que `game_flow.py` devienne trop gros en V4

Mais aujourd’hui, ce n’est pas le cas.

---

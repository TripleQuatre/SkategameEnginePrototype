# 📁 Dossier `modes/`

## 🎯 Rôle global

Le dossier `modes/` contient la logique liée aux **modes de jeu**.

Son rôle est de permettre au moteur de supporter plusieurs structures de partie sans mélanger cette logique directement dans `engine/`.

👉 En V3, ce dossier est encore minimal, mais il pose une base importante pour l’évolution future.

---

## 📄 `base_mode.py`

### Rôle

Définit la **base commune** des modes.

Contient une interface minimale avec :

* une méthode `validate(...)`

### Position dans l’architecture

* utilisé par `GameEngine`
* sert de type commun pour `_load_mode()`

### Pourquoi c’est utile

Sans ce fichier, le moteur serait obligé de typer les modes avec des classes concrètes (`OneVsOneMode`, etc.), ce qui compliquerait l’ajout de nouveaux modes.

Avec `BaseMode`, le moteur reste indépendant de l’implémentation précise du mode.

### Évolution probable

Plus tard, cette base pourra accueillir d’autres méthodes communes, par exemple :

* validation de configuration spécifique
* sélection des défenseurs
* règles de rotation
* adaptation du flow

### Améliorations possibles

* aucune urgente pour la V3
* il faut simplement éviter de la complexifier trop tôt

---

## 📄 `one_vs_one.py`

### Rôle

Implémente le mode **1 contre 1**.

Responsabilité actuelle :

* vérifier que la partie contient exactement 2 joueurs

### Position dans l’architecture

* instancié par `GameEngine`
* validé au démarrage de la partie

### Pourquoi c’est utile même si c’est simple

Ce fichier peut sembler léger, mais il a déjà un vrai rôle :

* il confirme que le mode est bien séparé
* il permet de brancher d’autres modes plus tard
* il évite de mettre cette validation directement dans `engine/`

### Évolution probable

Le mode 1v1 pourrait plus tard :

* adapter le flow
* définir ses propres comportements d’historique
* influencer certains choix de règles ou d’affichage

### Améliorations possibles

* aucune nécessaire maintenant
* le fichier est à la bonne taille pour la V3

---

## 🧠 Conclusion du dossier `modes/`

Le dossier `modes/` est encore minimal, mais il est **architecturalement important**.

Il montre que :

* le moteur est prêt à supporter plusieurs modes
* la logique de mode n’est pas enfouie dans `engine/`

👉 C’est une base propre pour la suite.

---

## 🔎 À retenir

* `base_mode.py` = interface commune des modes
* `one_vs_one.py` = implémentation actuelle du mode 1v1

---

Quand d’autres modes arriveront, il faudra décider :

* ce qui reste dans `GameFlow`

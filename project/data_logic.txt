# 🔄 Circulation globale des données dans SkateGame V3

## 🎯 Objectif

Ce document explique **comment une action utilisateur traverse tout le système**, depuis l’interface jusqu’au moteur, puis revient sous forme de résultat affiché.

👉 C’est la clé pour comprendre comment toutes les couches du projet interagissent ensemble.

---

## 🧭 Vue d’ensemble

```text
Utilisateur
   ↓
Interface (CLI / GUI)
   ↓
Controller
   ↓
Engine (GameEngine → GameFlow)
   ↓
Rules + Validation
   ↓
Core (State + Events)
   ↓
History
   ↓
Interface (affichage)
```

---

## 🔁 Exemple concret : déroulement d’un tour

Prenons un cas simple :
👉 un joueur entre un trick et le confirme

---

## 1. Interface → Controller

### Exemple (GUI ou CLI)

L’utilisateur :

* entre un trick
* clique sur “Confirm trick”

👉 L’interface appelle :

```python
controller.start_turn(trick)
```

---

## 2. Controller → Engine

Le `GameController` :

* ne fait aucune logique métier
* transmet simplement l’appel

```python
self.engine.start_turn(trick)
```

👉 Il joue un rôle de **relai propre**

---

## 3. Engine → GameFlow

Le `GameEngine` délègue à :

```python
GameFlow.start_turn(state, trick)
```

👉 `GameFlow` est le chef d’orchestre

---

## 4. Validation de l’action

Avant toute modification :

```python
ActionValidator.validate_start_turn(...)
```

👉 Empêche :

* tricks invalides
* actions au mauvais moment
* incohérences de flow

---

## 5. Application des règles

Le moteur utilise :

```python
RulesRegistry
```

pour accéder à :

* `special_rules`
* `scoring_rules`

Exemples :

* normaliser le trick
* vérifier qu’il n’a pas déjà été validé

---

## 6. Mise à jour du GameState

Le moteur modifie l’état :

* `current_trick`
* `defender_indices`
* `defense_attempts_left`
* etc.

👉 Tout passe par `GameState`

---

## 7. Création d’événements

Chaque action produit des événements :

```python
Event(name="turn_started", payload={...})
```

👉 Ces événements sont :

* ajoutés à `History`
* utilisés pour reconstruire le match

---

## 8. Résolution de la défense

Quand un joueur répond :

```python
controller.resolve_defense(success)
```

Cela déclenche :

* `TurnResolver`
* `ScoringRules`
* `EndConditions`

---

## 9. Mise à jour du score et élimination

Selon le résultat :

* ajout de lettres
* mise à jour du score
* désactivation d’un joueur si éliminé

---

## 10. Vérification de fin de partie

```python
EndConditions.is_game_finished(...)
```

👉 Si oui :

* événement `game_finished`
* identification du gagnant

---

## 11. Retour vers l’interface

L’interface récupère :

```python
state = controller.get_state()
```

Et affiche :

* score
* état du tour
* messages
* historique

---

## 12. Reconstruction de l’historique

Pour afficher le tableau :

```python
state.history.build_rows()
```

👉 Transforme les événements en lignes lisibles

---

# 🧠 Ce qu’il faut retenir

## 1. Le moteur est indépendant

* aucune dépendance à la GUI ou CLI
* testable en isolation

---

## 2. Le flux est unidirectionnel

* entrée → traitement → sortie
* pas de logique “cachée” dans l’interface

---

## 3. Les événements sont centraux

* tout passe par eux
* ils permettent :

  * historique
  * debug
  * futur export

---

## 4. `GameState` est le cœur

* source unique de vérité
* partagé par tout le système

---

## 5. Les responsabilités sont claires

| Couche      | Rôle                    |
| ----------- | ----------------------- |
| Interface   | interaction utilisateur |
| Controller  | relais                  |
| Engine      | orchestration           |
| Rules       | logique métier          |
| Core        | données                 |
| History     | mémoire du match        |
| Persistence | sauvegarde              |

---

# 🚀 Conclusion

Le projet suit un flux clair :

> Une action utilisateur traverse un pipeline structuré,
> modifie un état central,
> produit des événements,
> puis revient sous forme d’affichage.

👉 Cette architecture permet :

* une grande lisibilité
* une forte testabilité
* une excellente extensibilité

---

**Confiance : 1.00**

**Caveat :**

* ce flux est optimisé pour la V3 (1v1), mais reste suffisamment générique pour évoluer vers des modes plus complexes sans être refactorisé.

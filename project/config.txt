# 📁 Dossier `config/`

## 🎯 Rôle global

Le dossier `config/` contient toutes les **données de configuration du jeu**, c’est-à-dire :

* les paramètres nécessaires pour lancer une partie
* les règles modifiables du jeu
* la base d’un futur système de presets / configurations multiples

👉 C’est une couche **déclarative** : elle ne contient pas de logique, uniquement des structures de données.

---

## 📄 `match_parameters.py`

### Rôle

Représente les **paramètres nécessaires pour démarrer une partie**.

Contient :

* `player_ids`
* `mode_name`
* `rule_set`

👉 C’est le point d’entrée principal pour créer une partie.

### Position dans l’architecture

* utilisé par le **controller**
* injecté dans le **GameEngine**
* validé par `ConfigValidator`

### Évolution probable

* ajout de nouveaux paramètres :

  * seed aléatoire
  * variantes de gameplay
  * options de match (best of, durée, etc.)

### Améliorations possibles

* typage plus strict de `mode_name` (Enum plus tard)
* validation plus poussée (ex : nombre de joueurs selon mode)

---

## 📄 `rule_set_config.py`

### Rôle

Définit les **règles du jeu modifiables**.

Contient actuellement :

* `letters_word`
* `elimination_enabled`
* `defense_attempts`

👉 C’est le cœur des règles configurables du Game of Skate.

### Position dans l’architecture

* utilisé par les **rules**
* consommé par le **engine**
* validé par `ConfigValidator`

### Évolution probable

* ajout de nouvelles règles :

  * système de score alternatif
  * variantes de pénalité
  * règles spéciales (bonus, handicaps…)

### Améliorations possibles

* séparation future si les règles deviennent très nombreuses
* documentation interne des règles (docstring plus détaillée)

---

## 📄 `game_config.py`

### Rôle actuel

Fichier **réservé / préparatoire**.

Contient :

* `min_players`
* `max_players`
* parfois des éléments redondants avec `rule_set_config`

👉 Il n’est pas encore pleinement utilisé dans la V3.

---

### Rôle futur (très important)

Ce fichier est destiné à devenir une **couche de configuration globale**, qui pourrait :

* regrouper plusieurs sous-configs
* définir des presets complets
* gérer des modes de jeu plus complexes

Exemples futurs :

* config “casual”
* config “competitive”
* config “custom user”

---

### Position dans l’architecture (future)

* au-dessus de `match_parameters`
* pourrait devenir un point d’entrée unique pour configurer une partie

---

### Évolution probable

* centralisation des configs
* système de presets
* liaison avec la persistance (sauvegarde de configs)

---

### Améliorations possibles (maintenant)

Deux options valides :

👉 Option 1 (recommandée V3) :

* laisser le fichier tel quel
* ajouter un commentaire expliquant son rôle futur

👉 Option 2 :

* alléger le contenu pour éviter la redondance actuelle

---

Le dossier est :

* simple

* bien positionné dans l’architecture

👉 Il joue déjà son rôle pour la V3
👉 et prépare correctement les évolutions futures
